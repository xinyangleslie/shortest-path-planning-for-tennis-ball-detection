# 网球检测跨场景泛化实验报告

## 1. 实验概述

本报告总结 YOLOv26 网球检测模型的跨场景泛化实验。实验目标是评估不同训练数据组合对模型在公开数据、室外真实球场、室内真实球场之间泛化能力的影响，并为下一阶段 Ubuntu 视频 tracking 测试筛选候选模型。

本实验包含：

- 3 种模型规模：`yolo26n`、`yolo26s`、`yolo26m`
- 4 种训练集组合：`R`、`RC1`、`RC2`、`RC1C2`
- 6 种测试集：`R`、`C1`、`C2`、`RC1`、`RC2`、`RC1C2`
- 共 72 组 evaluation

主要结论是：训练数据多样性比单纯增大模型规模更重要。由 Rainbow、Court1、Court2 共同组成的 `RC1C2` 训练集具有最稳定的跨场景表现。

## 4. 主要发现

### 4.1 C1 室外场景最难泛化

只使用 Rainbow 训练时，模型在 C1 上的平均 mAP50 只有 0.1049。即使加入 C2 构成 RC2，模型在 C1 上也只有 0.1324 mAP50。

这说明室内 Court2 数据不能替代室外 Court1 数据。如果最终系统需要在室外球场运行，训练集中必须包含 Court1 或类似室外真实场景。

### 4.2 C2 与 Rainbow 更接近

只使用 Rainbow 训练时，模型在 C2 上仍能达到 0.8865 mAP50。加入 C2 后，RC2 在 C2 上达到 0.9919 mAP50 和 0.8609 mAP50-95。

这说明 C2 与 Rainbow 的视觉分布更接近，而 C1 与 Rainbow 的差异更大。

### 4.3 RC1C2 是最稳定的训练组合

`RC1C2` 同时包含公开数据、室外真实场景和室内真实场景。它不一定在每一个单独测试集上都是最高，但它在所有测试集上表现最均衡。

因此，`RC1C2` 是最适合作为通用部署和下一阶段 Ubuntu tracking 测试的训练组合。

### 4.4 模型规模影响有限

`yolo26n_RC1C2`、`yolo26s_RC1C2`、`yolo26m_RC1C2` 的平均 mAP50 差距很小：

- `yolo26n_RC1C2`: 0.8255
- `yolo26s_RC1C2`: 0.8333
- `yolo26m_RC1C2`: 0.8364

medium 模型略高，但 small 模型已经非常接近 medium，同时预计资源占用更低。

### 4.5 定位精度仍然是瓶颈

mAP50-95 明显低于 mAP50，尤其在 C1 和 RC1 上更明显。这说明模型通常可以检测到网球，但 bounding box 的精确程度还不够稳定。

这会影响后续任务，例如：

- 视频 tracking
- 轨迹估计
- 基于深度的 3D 坐标估计
- 机器人导航与路径规划

下一阶段需要重点观察视频中的 tracking 稳定性，而不能只看检测 mAP。

## 5. Ubuntu Tracking 推荐模型

下一阶段将在 Ubuntu 中测试视频 tracking 表现。推荐测试以下三个模型：

| 模型 | 用途 | 权重路径 |
|---|---|---|
| yolo26n_RC1C2 | 轻量实时基线 | `experiment/runs/yolo26n_RC1C2/weights/best.pt` |
| yolo26s_RC1C2 | 推荐主力模型 | `experiment/runs/yolo26s_RC1C2/weights/best.pt` |
| yolo26m_RC1C2 | 精度上限模型 | `experiment/runs/yolo26m_RC1C2/weights/best.pt` |

### 5.1 yolo26n_RC1C2

该模型权重最小，约 5.4 MB，预计 FPS 最高、显存占用最低。它适合作为轻量实时基线。

### 5.2 yolo26s_RC1C2

这是当前最推荐的主力模型。它的精度接近 `yolo26m`，但资源消耗预计更低，适合优先进入 Ubuntu tracking 测试。

### 5.3 yolo26m_RC1C2

该模型平均 mAP50 最高，但权重约 44.0 MB，资源占用最大。它适合作为精度上限，用来判断额外计算成本是否值得。

## 6. Ubuntu Tracking 建议记录指标

在 Ubuntu 中应使用同一段视频分别测试三个模型，并记录：

| 指标 | 含义 |
|---|---|
| FPS | 实时视频处理帧率 |
| ms/frame | 单帧平均耗时 |
| GPU memory | 显存占用，可用 `nvidia-smi` 查看 |
| GPU utilization | GPU 使用率 |
| CPU usage | CPU 占用 |
| RAM usage | 内存占用 |
| detection count | 检测到的网球数量 |
| missed frames | 球存在但未检测到的帧数 |
| false positives | 非网球被误检为网球的次数 |
| tracking lost count | tracking 中断次数 |
| trajectory stability | 轨迹是否平滑、连续 |

最终模型选择应该综合考虑：

```text
检测精度 + tracking 稳定性 + 实时性能 + 资源占用
```

## 7. 下一阶段测试计划

建议测试顺序：

1. 首先测试 `yolo26s_RC1C2`
   - 作为平衡型主力候选模型
   - 检查 FPS 是否满足实时需求
   - 检查 tracking 是否稳定

2. 然后测试 `yolo26n_RC1C2`
   - 作为轻量模型基线
   - 观察 FPS 提升幅度
   - 判断精度下降是否可以接受

3. 最后测试 `yolo26m_RC1C2`
   - 作为精度上限模型
   - 观察资源占用和帧率下降
   - 判断 medium 模型是否值得部署

如果 `yolo26s_RC1C2` 的 FPS 和 tracking 稳定性都满足要求，建议将其作为最终部署模型。如果 Ubuntu 设备算力不足，则考虑使用 `yolo26n_RC1C2`。如果 GPU 资源充足且对漏检特别敏感，可以考虑 `yolo26m_RC1C2`。

---

## 8. 总结

当前实验表明：

- 数据集多样性是跨场景泛化的关键因素
- C1 室外数据对室外球场检测非常重要
- RC1C2 是最稳定的训练组合
- RC1C2 下的三个模型都是下一阶段 tracking 的合理候选
- `yolo26s_RC1C2` 是当前最平衡、最值得优先测试的模型

下一阶段应重点比较 `yolo26n_RC1C2`、`yolo26s_RC1C2`、`yolo26m_RC1C2` 在真实视频 tracking 中的 FPS、资源占用和轨迹稳定性。
